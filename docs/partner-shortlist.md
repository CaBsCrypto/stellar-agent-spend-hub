# MCP/API Partner Discovery Shortlist

These are interview candidates, not confirmed integrations or partnerships.
Priority is based on having an official MCP or agent-facing API and a resource
that can plausibly be priced per call.

| Candidate | Paid resource hypothesis | Why interview |
| --- | --- | --- |
| Browserbase | Browser session or task | Official hosted MCP browser surface |
| Firecrawl | Crawl, scrape or research job | Official MCP server and usage-based resources |
| Tavily | Search or extraction request | Official remote MCP search tools |
| Apify | Actor run or dataset result | Official Streamable HTTP MCP server |
| Bright Data | Search, unlocker or scraping result | Separate Web MCP surface for live web data |
| Alchemy | Blockchain data or simulation call | Official MCP and agent payment direction |
| Exa | Search or research result | Agent-oriented search API; MCP fit to validate |
| QuickNode | RPC or marketplace add-on call | Agent/x402 direction to validate directly |
| Pinata | Storage or gateway operation | API-first digital resource; MCP demand to validate |
| Cloudflare | Worker AI or browser task | Usage-metered infrastructure; MCP fit to validate |

## Interview questions

1. Which resource is naturally sold per request rather than by monthly seat?
2. Would an agent-specific payment rail reduce signup or billing friction?
3. What maximum amount can be authorized without human review?
4. Which receipt fields are required for support and reconciliation?
5. Can the resource be prepared before charging to avoid paid failures?
6. Would Stellar USDC testnet evidence justify a technical pilot?
